# Data Model: EF Core Scaffold and Health API

**Feature Branch**: `001-efcore-health-api`
**Date**: 2026-02-21

## Health Check Response

A simple value object representing the `/health` endpoint response.
Not persisted to the database.

| Field | Type | Description |
|-------|------|-------------|
| Status | string | "healthy" or "unhealthy" |
| DatabaseVersion | string? | PostgreSQL version string (null when unhealthy) |
| ServerTime | DateTimeOffset? | Current DB server timestamp (null when unhealthy) |
| Error | string? | Error description (null when healthy) |

**Notes**:
- This entity is a response DTO, not a database entity.
- Per Principle VI, this MUST be an immutable `record` type.
- Status values are a closed set: consider using an enum
  or string constants.

## Post-Scaffold Adjustments

Per constitution principles, the scaffolded entities require:

1. **Principle VI compliance**: The Health Check Response DTO
   MUST be a `record`. Scaffolded EF entities remain as classes
   (EF Core change tracking requires mutable properties — this
   is a justified framework constraint).
2. **Principle III compliance**: The `DbContext` and entities
   live in the infrastructure/data access layer. Domain logic
   MUST NOT depend on them directly.
3. **Principle V compliance**: All entity classes MUST have XML
   documentation comments describing their purpose and
   relationships.
